KEY DISCUSSION POINTS
─────────────────────────────────────────────────────────────────────────────────

1. DEMO SCOPE & APPROACH
   • Live demo using es2os (Elasticsearch → OpenSearch analyzer)
   • Docker Compose setup with ES 7.17 + OpenSearch 2.19 + Logstash
   • Kind cluster optional (took 30 min to build, complex for stage)
   • Migration Assistant referenced as production architecture (not live demo)

2. DENSE_VECTOR → KNN_VECTOR TRANSFORMATION
   • Core migration issue: ES uses dense_vector, OS uses knn_vector
   • es2os detects this automatically (VEC001 critical issue)
   • Migration Assistant auto-applies during metadata migration
   • Requires HNSW algorithm config in OpenSearch
   • This is the "headline" transformation that shows why tool is needed

3. THREE MIGRATION SCENARIOS COVERED
   • Phase 1: Analysis (es2os report) - "What will break?"
   • Phase 2: Transformation & Backfill (Logstash) - "How to move it?"
   • Phase 3: Cutover & Zero-Downtime (Dual-write pattern) - "How to switch?"

4. ZERO-DOWNTIME MIGRATION PATTERN
   • Dual-write: App writes to BOTH ES 7.17 AND OpenSearch 2.19
   • Logstash backfills historical data in parallel
   • No downtime required - ES stays operational throughout
   • Easy rollback: switch app connection string if issues detected

5. DATA CONSISTENCY & ERROR HANDLING
   • Logstash handles batching, retries, offset tracking automatically
   • ILM → ISM policy manual conversion (architectural decision required)
   • _source disabled indices need enableSourcelessMigrations flag
   • Migration Assistant built-in validation catches blockers

6. TOOL SELECTION: LOGSTASH VS BEATS
   • Logstash: Better for migration (transformations, bulk one-time)
   • Beats: Better for continuous shipping (lightweight agents)
   • For this talk: Logstash with docker-compose (simple, reproducible)

7. KIND CLUSTER EXPERIENCE
   • Setup: 30 min to build all images from source (JDK 11-17 required)
   • Worked but complex for stage demo (too many moving parts)
   • Better for post-talk hands-on lab, not live 40-min presentation

8. MIGRATION ASSISTANT FINDINGS
   • Kubernetes/EKS-native (not lightweight for local demo)
   • Uses Snapshot + Metadata Migration + Document Backfill workflow
   • Has approval gates (need human decision between phases)
   • Auto-detects same issues as es2os (confirms tool compatibility)

9. TESTING INSIGHTS
   • ES 7.10 dense_vector without index:true causes transformer issues
   • Migration Assistant needs similarity field for proper knn_vector generation
   • Network connectivity must match Kubernetes internal DNS (es-source, os-target)
   • SearchGuard ES requires HTTPS + credentials for queries

KNOWN LIMITATIONS & GOTCHAS
─────────────────────────────────────────────────────────────────────────────────

1. Dense vector mapping incomplete in ES 7.10
   - Solution: Ensure index:true and similarity:cosine in mapping

2. Migration Assistant ES 7.10 transformer bug
   - Solution: Skip product_embeddings, migrate manually post-backfill
   - Or use indexAllowlist to exclude from metadata migration

3. Logstash network connectivity
   - Must be on same docker network as ES and OS (es2os_migration-net)
   - Container names: es-source:9200, os-target:9200

4. SearchGuard authentication
   - OpenSearch test cluster uses HTTPS + admin:admin
   - Requires: allowInsecure, authConfig with secretName

5. ILM policy manual conversion
   - es2os flags it as "manual"
   - Migration Assistant also requires human review
   - Not an auto-fixable transformation
